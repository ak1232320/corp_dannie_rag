# Улучшения сверх базового пайплайна

Базовый эталон — лексический поиск (TF-IDF + cosine) и экстрактивный «ответ»
склейкой фрагментов, без LLM. Реализованы **три улучшения**.

---

## 1. Гибридный поиск (TF-IDF + семантические эмбеддинги)

**Суть.** К лексическому поиску добавлен семантический: чанки кодируются моделью
`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` (384-dim, L2-норма).
При запросе считаются два косинуса — лексический (TF-IDF) и семантический
(эмбеддинги), каждый нормируется min-max по кандидатам и смешивается:
`score = α·sem + (1−α)·lex`, `α = HYBRID_ALPHA = 0.5`.

**Зачем.** TF-IDF не ловит синонимы/перефраз и **не работает между языками**.
Эмбеддинги добавляют смысл; гибрид устойчив к разным типам запросов.

**Файлы.** [`app/embedder.py`](../app/embedder.py),
[`app/hybrid_retriever.py`](../app/hybrid_retriever.py),
интеграция в [`scripts/build_index.py`](../scripts/build_index.py) (сохранение
`embeddings.npy`) и [`app/main.py`](../app/main.py) (переключатель + показ
компонент `lex`/`sem`).

**Как проверено.** `tests/test_hybrid.py` (нормировка, слияние, ранжирование),
`scripts/evaluate.py`. Результат — кросс-язычные запросы (RU → EN-корпус):

| ru_paraphrase hit@3 | TF-IDF | Hybrid |
|---|---|---|
| | **0.125** | **1.000** |

**Важный нюанс.** min-max теряет абсолютную релевантность (топ всегда ≈1), поэтому
отказ от офтопа опирается не на смешанный скор, а на «сырые» компоненты — см. улучшение про отказ ниже и `is_relevant()`.

---

## 2. Оценка качества (retrieval@k / hit-rate / MRR)

**Суть.** Автоматические метрики вместо ручной проверки демо-вопросами. Набор из
57 размеченных запросов 4 типов: `title` (заголовок → его doc_id), `paraphrase`
(EN-описание заболевания → любой doc заболевания), `ru_paraphrase` (то же
по-русски), `negative` (офтоп → ожидается отказ). Сравниваются **TF-IDF и гибрид**.

**Зачем.** Воспроизводимая, объективная оценка; ловит регрессии; показывает, где
именно помогает семантика.

**Файлы.** [`scripts/evaluate.py`](../scripts/evaluate.py),
[`eval/eval_queries.json`](../eval/eval_queries.json) (генерируется),
[`eval/doc_conditions.json`](../eval/doc_conditions.json), `tests/test_eval.py`.

**Как проверено.** `uv run python scripts/evaluate.py`:

| набор | TF-IDF hit@3 | Hybrid hit@3 |
|---|---|---|
| title | 1.000 | 1.000 |
| paraphrase (EN) | 1.000 | 1.000 |
| ru_paraphrase | 0.125 | **1.000** |
| negative (отказ) | ✗ | **✓** |
| **итого** | 0.875 | **1.000** |

Дополнительно: пороги отказа калиброваны по реальным распределениям скоров.
На этом корпусе **лексические скоры офтопа высоки** (общие слова how/do/home дают
TF-IDF до ~0.38 и пересекаются с доменом 0.45–0.56), а **семантические — нет**
(офтоп ≤0.37, домен ≥0.58). Поэтому в гибриде релевантность определяется
семантикой (`SEM_MIN_SCORE=0.45`), и офтоп («How do I cook borscht?»)
корректно отклоняется, чего чистый TF-IDF не умеет.

## 3. LLM-генерация ответа (OpenRouter)

**Суть.** Вместо склейки фрагментов — связный ответ LLM строго по найденному
контексту, со ссылками на источники. OpenRouter (OpenAI-совместимый API),
системные правила «отвечай только по контексту, иначе откажись».

**Зачем.** Читабельный ответ на языке вопроса; меньше «простыни» из чанков.

**Файлы.** [`app/llm.py`](../app/llm.py) (клиент OpenRouter),
[`app/prompts.py`](../app/prompts.py) (`LLM_PROMPT_TEMPLATE`, `SYSTEM_RULES`),
ветка в [`app/generator.py`](../app/generator.py), тумблер в `app/main.py`.

**Безопасный фолбэк.** Если `OPENROUTER_API_KEY` не задан или вызов упал —
`llm.generate()` возвращает `None`, и ответ строится экстрактивно. **Приложение
всегда работает.** Режим (`llm` / `extractive` / `refusal`) показывается в ответе.

**Как проверено.** `tests/test_generation.py` (graceful fallback без ключа, отказ,
экстрактив), ручной прогон `scripts/check_generator.py`. Конфигурация модели —
`OPENROUTER_MODEL` (по умолчанию `openai/gpt-4o-mini`).
